# Кейсы расследований

Цель: **всёобъемлющие заметки в Obsidian** — выгрузка контекста расследования, чтобы в будущем искать в vault, а не в интернете.

## Показ пользователю

- Ответ MCP — **внутренний**. В чат — только `userSummaries` / `userMessage` своими словами.
- **Запрещено** вставлять JSON, `matches`, `obsidian`, `agent_action`, полный кейс.
- Кейс с другой `configurationName` — явно «только ориентир», не применять вслепую.

## Обязательный порядок

1. **Сначала** — симптом от пользователя → `onec_declare_symptom(symptom=...)`.
2. **Затем** — `onec_search_cases(query=...)` по зафиксированному симптому.
3. Если `mustReviewBeforeInvestigation: true` (есть кейсы **той же** конфигурации) — `onec_get_case` для себя; пользователю — краткое резюме.
3. **При приближении к решению** (гипотезы сходятся, чек-лист сработал, пользователь подтверждает направление):
   - `onec_save_case(status=draft, ...)` — **полный** контекст;
   - `onec_obsidian_save_session(...)` — заметка Sessions с ходом диалога.
   - `onec_investigation_status` — запомнить `case_id` и `session_relative_path`.
4. **При финальном решении** — `onec_save_case(case_id=..., status=final, correct_solution=...)`.
5. **При доп. вопросах / новых задачах в той же сессии** — **дополнять**, не создавать дубликаты:
   - `onec_save_case(case_id=..., additional_notes=..., methods_applied=..., hypotheses=...)`
   - `onec_obsidian_append_session(content=..., section=...)`

## Когда сохранять draft

- Найдена вероятная причина, но ещё не всё проверено.
- Пользователь согласился с гипотезой.
- Собраны важные факты из ИТС, метаданных, кода, ИБ — **до потери контекста**.

## Поля onec_save_case (всеобъемлющий кейс)

| Параметр | Смысл |
|----------|--------|
| `symptom` | Симптом / задача |
| `context_summary` | Конфигурация, версия, документ, пользователь, ограничения |
| `investigation_path` | Краткий нарратив: что делали по шагам |
| `methods_applied` | Через `;`: ИТС, метаданные, onec_query, XML-код, форумы, чек-лист |
| `hypotheses` | Гипотезы и уровень уверенности |
| `wrong_approach` | Что не сработало и почему |
| `correct_solution` | Итоговое решение (обязательно при `status=final`) |
| `sources_used` | Через `;`: статьи ИТС, кейсы, пути к модулям |
| `objects_used` | Через `;`: объекты метаданных |
| `queries_used` | Через `;`: запросы к ИБ |
| `checklist` | Что проверять при повторении |
| `tags` | Через запятую |
| `additional_notes` | Дополнение в хронологию (новые вопросы пользователя) |
| `case_id` | Для обновления существующего кейса |
| `status` | `draft` или `final` |

## Структура заметки в Obsidian Cases/

Помимо трёх классических блоков (симптом, неверный путь, решение):

- Контекст
- Ход расследования + применённые способы
- Гипотезы
- Источники
- Объекты и запросы
- Чек-лист
- Хронология дополнений

## Пример (draft при приближении к решению)

```
onec_save_case(
  symptom="Остатки не сходятся после реализации",
  status="draft",
  context_summary="УТ 11.5, организация X, документ Реализация №123 от ...",
  investigation_path="1. Кейсы → 2. Метаданные регистров → 3. ИТС по передаче на комиссию → 4. Чек-лист в ИБ",
  methods_applied="onec_search_cases;onec_metadata_object;onec_its_fetch;чек-лист пользователю",
  hypotheses="Движения ушли в ТоварыОрганизаций, а смотрели ТоварыНаСкладах — высокая уверенность",
  wrong_approach="Искали только ТоварыНаСкладах",
  sources_used="ИТС: Передача товаров на комиссию; РегистрНакопления.ТоварыОрганизаций",
  objects_used="РегистрНакопления.ТоварыОрганизаций;Документ.РеализацияТоваровУслуг",
  checklist="Остатки ТоварыОрганизаций на дату документа; субконто комиссионер"
)
onec_obsidian_save_session(summary="Расхождение остатков после реализации", transcript_markdown="...")
```

## Дополнение при уточнении

```
onec_obsidian_append_session(
  section="Уточнение пользователя",
  content="Спросил про возврат от комиссионера — добавлена проверка документа Возврат..."
)
onec_save_case(
  case_id="<из onec_investigation_status>",
  additional_notes="Пользователь уточнил сценарий с возвратом от комиссионера",
  methods_applied="onec_metadata_search ВозвратТоваровОтКлиента",
  status="draft"
)
```

## Метаданные

Перед поиском объектов — `onec_metadata_status`. Не смешивать конфигурации.
