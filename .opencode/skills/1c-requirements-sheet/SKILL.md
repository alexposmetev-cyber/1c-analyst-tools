# Лист требований



Шаблон: `templates/Лист_требований.md` (в корне `1c-analyst-tools`).



## Когда применять



Пользователь просит **лист требований** / ЛТ / оформление задачи, присылает **транскрипт** или описание доработки.



Лист требований **не заменяет** расследование бага — уточни, если неясно.



## Изоляция конфигураций (обязательно)



**Нельзя** переносить объекты, регистры, механизмы и «типовые решения» из другой конфигурации.



- Контекст привязан к **текущей базе** (`.Obsidian/{ИмяБазы}/`) и **имени конфигурации** из connect/метаданных.

- Если пользователь переключил базу — `onec-data_onec_disconnect` → новый `onec-data_onec_connect`; не используйте кэш/кейсы другой ИБ.

- В §4 и ответе указывайте только объекты, подтверждённые метаданными **этой** базы.



## Запрещено



- Выдумывать факты, которых нет во входе.

- Оставлять §7 пустым или с пометкой «по согласованию» без цифр.

- Писать §4 одним абзацем без подразделов 4.1–4.4.

- Пропускать §2.1 «Ценность для заказчика».

- В §4 — без лишнего жаргона разработчика (см. стиль ниже).

- Передавать `platform_version` в `onec-data_onec_connect` — платформа подбирается по ИБ и `1cestart.cfg` автоматически.

## Фаза 0 — Контекст до черновика ЛТ (обязательно)



Сразу после получения описания задачи:



### 1. Obsidian



```

onec-data_onec_obsidian_prepare_requirements(task_description=..., configuration_name=...)

```



Инструмент возвращает:



| Блок | Действие агента |

|------|-----------------|

| `cases` / `pastRequirements` | Было ли похожее — учесть решения, не повторять ошибки |

| `handbooks` | Есть ли `Справочники` — искать типовой функционал |

| `configurationIsolation` | Соблюдать весь цикл ЛТ |

| `bitmedic` | Если отраслевая БИТ — [info.bitmedic.ru](https://info.bitmedic.ru/) |



При необходимости отдельно: `onec-data_onec_obsidian_search_handbooks(query=...)`.



**Если в Справочниках найдено** — в §4 опирайтесь на типовой функционал; доработка может не понадобиться.



**Если справочников нет** — явно: контекст дополняется из ИТС/интернета/кода; при необходимости предложите положить выжимку в `Справочники/`.



### 2. Поиск вне Obsidian (если в vault мало)



Порядок:



1. **`onec-data_onec_its_search`** / **`onec-data_onec_its_fetch`** — платформа и конфигурация.

2. Отраслевые **БИТ** (Управление медицинским центром, Стоматология, Айболит, Красота, Фитнес):

   - **`onec-data_onec_bitmedic_guidance`** → поиск на [info.bitmedic.ru](https://info.bitmedic.ru/) (`site:info.bitmedic.ru …`).

   - На портале не всегда есть ответ по доработкам — не выдавать за факт без проверки в метаданных.

3. **`onec-data_onec_web_search_forums`** — только если ИТС/БИТ не дали ориентира; помечать как гипотезу.



### 3. Разработка в задаче



Если задача предполагает **доработку кода** (новая логика, перехват, отчёт, интеграция):



- Skill **`1c-config-sources`**: **спросите**, есть ли уже XML-исходники → `onec-data_onec_config_sources_register`.

- Иначе после **`onec-data_onec_connect`** — `onec-data_onec_dump_config(mode=partial, objects=...)` или Full по согласию.

- **`onec-data_onec_metadata_*`** — имена объектов; **`onec-data_onec_config_search_code`** / **`onec-data_onec_config_read_module`** — факты из BSL.

- Код и метаданные — **только** текущей базы/каталога исходников.

## Обязательное содержание ЛТ



| Раздел | Требование |

|--------|------------|

| **§2.1 Ценность для заказчика** | Отдельный подраздел: зачем бизнесу, что получит после внедрения, как изменится работа; по возможности — измеримый эффект |

| **§4 Способ реализации** | Подразделы **4.1–4.4**: результат для пользователя, пошаговый план, правила/исключения, что не входит. Достаточно подробно, чтобы заказчик понял путь к результату |

| **§7 Оценка трудозатрат** | Таблица **всегда заполнена**: строки = этапы §3, часы по каждому, итого и стоимость (4950 руб./час). В черновике — предварительно; в final — уточнённо |

## Фаза A — Черновик ЛТ в чат



После фазы 0 оформите ЛТ по шаблону (§1–7):



- §2.1 — ценность для заказчика;

- §4 — предварительно, с подразделами 4.1–4.4 (Obsidian/ИТС/справочники);

- §7 — **предварительная** оценка часов по этапам §3 (даже без connect).



В конце:



1. **Основные мысли из текста**

2. **Уточнить у заказчика**

3. **Источники контекста** — кейсы, справочники, ИТС, bitmedic (кратко)

4. **`onec-data_onec_obsidian_save_requirements`** — `phase="draft"`



---



## Фаза B — С базой и/или исходниками (по согласию)



1. **Спросите путь к XML**, если ещё не спрашивали (`onec-data_onec_config_sources_register`).

2. При необходимости `onec-data_onec_connect` → `onec-data_onec_metadata_*` → `onec-data_onec_dump_config(partial)` если файлов нет.

3. `onec-data_onec_config_search_code` / `onec-data_onec_config_read_module` — факты для §4.2–4.3.

4. Обновить §1, §4, §7; **`onec-data_onec_obsidian_save_requirements`** — `phase="final"`.



**Не делать** без запроса: `onec-data_onec_query` к боевым данным.

## Стиль §4



**Подробность:** заказчик без опыта 1С должен понять цепочку «сейчас → что сделаем → как будет работать».



Хорошо:



- «В печатной форме титульного листа карты пациента появится блок „Параметры осмотра“: список названий и значений из приёма с нужным шаблоном осмотра.»

- «На канбане „Мои продажи“ под сроком на этапе появится номер последнего заказа по интересу; если заказов несколько — „номер + N“.»



Плохо:



- «Перехватить ОбработкаПроведения …»

- Один абзац без шагов и без 4.1–4.4.



Допустимо: *(документ ЗаказКлиента)* после понятного названия.



**§7:** сумма строк = итого; стоимость = итого × 4950. Укажите работы вне оценки, если объём ограничен.
