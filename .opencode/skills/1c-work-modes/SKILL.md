# Режимы работы аналитика

**Первый шаг каждой задачи** — определить режим. Не навязывай подключение к базе, если пользователь не может или не хочет его давать.

Спроси через `question`, если из сообщения неясно:

> «Есть прямой доступ к информационной базе (COM/клиент на этой машине)? Или вопрос общий / только по описанию и документации?»

---

## Режим A — **live** (есть доступ к ИБ)

**Когда:** пользователь может дать базу, логин, пароль; нужны фактические данные (номера документов, остатки, движения).

**Порядок:**

1. В новой чат-сессии `onec_welcome` уже сбросил старое подключение.
2. `onec_connection_status` → `onec_list_infobases` → **question** (база, пользователь, пароль) → `onec_connect`.
2. Далее skill **`1c-investigation-pipeline`**: метаданные → ИТС → чек-лист для пользователя → при эскалации `onec_read_module`.
3. `onec_query` — только по согласию пользователя.

**Приоритет источников:** ИТС → кейсы → код (конфигуратор) → данные ИБ (по запросу) → форумы.

Skills: `1c-connection`, `1c-investigation-pipeline`, `1c-web-research`, `1c-investigation`.

---

## Режим B — **offline** (нет доступа к ИБ)

**Когда:** удалённый клиент без COM, прод без выгрузки, NDA, «только опишу проблему», нет пароля/базы на этой машине, `onec_connect` недоступен или пользователь отказался.

**Не делай:**

- Не настаивай на `onec_connect` после отказа или неудачи.
- Не выдумывай цифры из документов, которых не видел.
- Не выдавай гипотезу за факт без пометки «предположение».

**Делай:**

1. Собери контекст через **question**: конфигурация и версия, симптом, текст ошибки, номер/дата документа, что уже пробовали, фрагменты кода/запроса, скрин, выгрузка.
2. `onec_search_cases` — по симптому и названию конфигурации.
3. Если в прошлой сессии был кэш метаданных той же конфигурации — можно `onec_metadata_search` / `onec_metadata_object` **только если** `onec_metadata_status` показывает `ready` и версия совпадает с описанием пользователя. Иначе — опирайся на имена объектов из слов пользователя и ИТС.
4. `onec_web_research_status` → ИТС / форумы (skill `1c-web-research`).
5. Ответ: **симптом → известные факты → гипотезы (с уровнем уверенности) → что проверить в базе**, когда доступ появится.

**Приоритет источников:** факты от пользователя → кейсы → кэш метаданных (если актуален) → ИТС → форумы.

Skill: `1c-investigation`, `1c-web-research`, `1c-cases`.

---

## Режим C — **research** (общий вопрос)

**Когда:** методология, платформа, «как устроено», стандарты, сравнение подходов, нет привязки к конкретным данным ИБ.

**Не делай:** `onec_connect`, `onec_query` — не нужны.

**Делай:**

1. Уточни конфигурацию и версию (для выбора базы ИТС и фильтра форумов).
2. `onec_web_research_status` → **question** (логин/пароль ИТС при необходимости) → `onec_its_configure`.
3. `onec_its_search` / `onec_its_fetch` — основной источник.
4. `onec_web_search_forums` — дополнение; **явно** помечай риск неактуальности для другой версии.
5. `onec_search_cases` — если есть похожий разбор.

**Приоритет источников:** ИТС → кейсы → форумы (как гипотезы).

Skill: `1c-web-research`.

---

## Смена режима

- Пользователь дал доступ к базе после offline → переключись на **live**, `onec_connect`.
- Live недоступен после попытки → честно перейди в **offline**, не повторяй connect без новых данных.
- Уточняющий вопрос без данных ИБ → оставайся в **research** / **offline**.

## Формулировки для пользователя

- Offline: «Без доступа к базе могу разобрать логику и типовые причины; для проверки фактов понадобится …»
- Research: «Отвечу по ИТС и открытым источникам; для вашей базы уточните версию конфигурации …»
- Live: «Подключусь read-only и проверю данные по документу …»
