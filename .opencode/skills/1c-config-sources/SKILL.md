# Исходники конфигурации (XML/BSL)

COM-подключение **не читает код модулей** — только метаданные (JSON-кэш) и данные (`onec_query`).  
Для разбора логики и обогащения ЛТ нужны **XML-исходники** с BSL.

## Первый шаг — всегда спросить пользователя

Через **question**:

> «Есть ли уже выгруженная конфигурация в файлах (Git, каталог разработки, XML рядом с проектом)? Если да — укажите полный путь к каталогу.»

| Ответ | Действие |
|-------|----------|
| **Есть путь** | `onec_config_sources_register(sources_path=...)` → `onec_config_sources_status` |
| **Нет / не знает** | Предложить `onec_dump_config` (нужен `onec_connect`) или ручную выгрузку в конфигураторе |
| **Отказ / нет доступа** | Оставаться в offline: ИТС, кейсы, гипотезы без кода |

**Не выгружайте** полную конфигурацию без согласия — Full может занять десятки минут.

---

## Регистрация готовых файлов

```
onec_config_sources_register(sources_path="D:\\repo\\src\\cf", label="...", configuration_name="...")
onec_config_sources_status
```

Каталог валиден, если есть `Configuration.xml` или подкаталоги `Catalogs` / `Documents` / `CommonModules`.

---

## Выгрузка из ИБ (если файлов нет)

1. `onec_connect` (без `platform_version`).
2. По метаданным выберите 3–10 релевантных объектов.
3. **`onec_dump_config(mode="partial", objects="Документ.X,ОбщийМодуль.Y")`** — быстрее Full.
4. Каталог регистрируется автоматически; смотрите `registered_source` в ответе.

Full (`mode="full"`) — только по явной просьбе пользователя.

Ручной скрипт (для пользователя, не предлагать агенту как основной путь): `scripts/Dump-1CConfigToFiles.ps1`.

## Анализ кода

| Инструмент | Назначение |
|------------|------------|
| `onec_config_read_module` | Текст модуля из XML (предпочтительно вместо `onec_read_module`) |
| `onec_config_search_code` | Поиск процедур, вызовов, строк по BSL |
| `onec_read_module` | Запасной: частичная выгрузка через конфигуратор (медленно) |

Порядок:

1. `onec_metadata_search` — имена объектов.
2. `onec_config_search_code(query="...")` — где реализована логика.
3. `onec_config_read_module(full_name, module_part)` — ключевые модули (1–3 за раз).

В ответах и ЛТ указывайте **факты из кода**: процедуры, условия, регистры — со ссылкой на файл (`relative_path`).

## Расследование ошибок

Если после ИТС и чек-листа причина не ясна:

1. Спросите про готовые XML (см. выше).
2. Зарегистрируйте или выгрузите partial по объектам из симптома.
3. `onec_config_search_code` + `onec_config_read_module`.
4. Гипотезы по коду + новый чек-лист для ИБ.
